# 소프트웨어 개발 방법론

## 1. 소프트웨어 공학

### 1. 상용 소프트웨어

1. 상용 소프트웨어 특징

   - **상업적 또는 판매를 목적**으로 개발
   - 라이선스를 이용해 배포되며 기술 지원을 보증
   - 소스코드 배포 X (복제, 변형 불가)

   > 핵심 기능을 축소한 프리웨어, 사용 기간을 제한하는 쉐어웨어, 광고를 포함하는 에드웨어 등도 이윤 창출을 목적으로 제작되기 때문에 상용 소프트웨어에 속함
   > {: .prompt-info }

2. 상용 소프트웨어 분류체계

   - **범용**: 시스템SW, 미들웨어, 응용SW 등
   - **특화**: 자동차, 항공, 교육, 물류 등

<br>

### 2. 응용 소프트웨어
   
1. 응용 소프트웨어 특징

   - 특정 업무 처리 목적으로 개발
   - **애플리케이션, 응용 솔루션 및 서비스** 등으로 불림

2. 응용 소프트웨어 분류체계

   - **기업용 소프트웨어**: 오피스웨어, ERP, SCM, BI, CRM 등
   - 영상 처리 소프트웨어: 영상 인식, 영상 편집, 스트리밍 등
   - CG/VR 소프트웨어: 3D 스캐닝과 프린팅, 모델링, 가상현실, 홀로그램 등
   - 콘텐츠 배포 소프트웨어: 콘텐츠 보호/관리/유통 등
   - 자연어 처리 소프트웨어: 정보 검색과 질의응답, 의사 결정 지원, 언어 분석 등
   - 음성 처리 소프트웨어: 음성 인식/합성/처리 등 

3. **기업용 소프트웨어**

   - 오피스웨어: 일반 업무용 (워드, 엑셀 등) 
   - ERP: 경영 활동 프로세스 통합 관리
   - SCM: 물류의 흐름 파악 및 지원
   - BI: 데이터를 활용하여  의사 결정 지원 
   - CRM: 고객 특성에 맞는 마케팅 활동 지원 

<br>

### 3. 시스템 소프트웨어
   
1. 시스템 소프트웨어 정의

   사용자가 쉽고 효율적으로 컴퓨터 시스템을 사용하도록 돕는 소프트웨어 <br>
   (일반적으로 하드웨어 제작사에서 제공되며, 운영체제와 같은 의미로 많이 쓰임)

2. 시스템 정의

   - 특정 업무를 수행하기 위해 중앙 처리 장치를 중심으로 구성된 소프트웨어, 하드웨어
   - 5가지 기능(입력, 처리, 출력, 제어, 피드백)을 수행하는 장치가 존재

3. 시스템 성능평가 기준

   - 처리능력: 단위 시간 내 작업 처리량
   - 반환 시간: 작업 의뢰부터 처리까지의 시간
   - 사용 가능도(가용성): 필요할 때 즉시 사용 가능한 정도 
   - 신뢰도: 주어한 문제를 정확하게 해결하는 정도 

4. 플랫폼 정의

   - 특정 시스템을 바탕으로 제공되는 운영체제 및 운영환경

   ```
   기준(관점)에 따른 플랫폼의 범위

   - 응용 프로그램 관점에서의 플랫폼 >> 윈도우즈, 안드로이드 등
   - 카카오 서비스 이용자 관점에서의 플랫폼 >> 카카오톡
   ```

5. 플랫폼 성능평가 기준

   - 사용률: 데이터 처리에 시스템 자원을 사용하는 정도
   - 응답 시간: 명령에 반응하는 시간 (처리시간과 다름)
   - 사용 가능도(가용성): 필요할 때 즉시 사용 가능한 정도
   - 정확성: 처리 결과가 기대한 값과 비교해서 정확한지 측정 

<br>

### 4. 소프트웨어 공학

1. 소프트웨어 공학 정의

   - **효율적인** 소프트웨어 개발을 위한 모든 수단과 도구
   - 개발의 품질과 생산성 향상을 목표로 연구하는 학문

   ```
   소프트웨어 공학의 목적

   - 필요한 비용과 기간의 예측
   - 하드웨어에 대한 소프트웨어의 상대적 비용 절감
   - 최신 하드웨어, 소프트웨어 기술 반영
   ```

2. 소프트웨어 공학의 기본 원칙

   - 현대적인 프로그래밍 적용을 지속적으로 적용
   - 품질에 대한 지속적인 검증
   - 개발 단계별 산출물에 대한 명확한 기록 유지 

<br>

### 5. 소프트웨어 개발 프레임워크

1. 모듈

   - 프로그램을 기능별로 분할하여 재사용 가능하도록 부품화한 것
   - 개발자가 프로그램의 기본 틀을 제공하고, 모듈을 사용해 기능을 구체화 (일반적 제어의 흐름)

2. 라이브러리

   - 관련 있는 모듈을 모아놓은 것 (툴킷)
   - 라이브러리에 존재하는 각 모듈이 반드시 독립적인 것은 아님

   ```
   라이브러리의 종류

   - 표준 라이브러리: 프로그래밍 언어에 내장
   - 외부 라이브러리: 별도의 설치를 통해 사용
   ```

3. 디자인 패턴

   - 특정 기능에 대한 문제해결을 위한 추상적 가이드라인 제시 <br>
   (특정 기능 구현 설계 가이드라인)

4. 소프트웨어 개발 프레임워크

   - 실질적인 개발의 틀을 제공 (디자인 패턴에 모듈의 장점 및 기능을 결합)
   - 프로그램의 기반, 구조를 잡아주는 코드의 모임으로 자연스럽게 특정 디자인 패턴을 유도
   - 프레임워크가 프로그램의 기본 틀을 제공하고, 개발자가 기능을 구체화 (제어의 역 흐름)
   - 검증된 프레임워크를 사용해 품질, 예산, 유지보수에 이점
   - 프로그램 구성 복잡도가 감소해 상호 운용성과 개발 및 변경이 용이 
   
   > 개발자가 프로그램의 흐름을 제어하는 것이 일반적이지만, 프레임워크는 이 흐름을 개발자 대신 주도하기에 개발의 난이도가 감소함 <br>
   > {: .prompt-info }


5. 소프트웨어 아키텍처

   - 다수의 프레임워크를 체계적으로 구성, 설명하는 구조체
   - 소프트웨어의 설계와 업그레이드를 통제하는 지침과 원칙 제공 

   > 디자인 패턴과 아키텍처 <br>
   > 아키텍처는 소프트웨어의 전체적인 구조 설계를 위한 방법을, 디자인 패턴은 특정 기능 구현을 위한 해결 방법을 제시
   > {: .prompt-info }

6. 컴포넌트

   - 모듈의 형태로 재사용 가능한 확장된 소프트웨어 블록
   - 컴포넌트를 결합하고, 인터페이스를 통해 서로에게 접근
   - 프레임워크 및 아키텍처가 적용되어 안정적
   - 모듈과 달리 컴포넌트는 자체만으로 소프트웨어의 역할 수행 가능

   ```
   협약에 의한 설계를 따를 경우 포함되는 조건

   - 선행조건: 컴포넌트 오퍼레이션 사용 전 참이여야하는 조건
   - 결과조건: 컴포넌트 오퍼레이션 사용 후 참이여야하는 조건
   - 불변조건: 컴포넌트 실행 중에 참이여야하는 조건
   ```

   > **협약에 의한 설계** <br>
   > - 인터페이스가 틀만 정했다면, 협약은 조건까지 정함
   > ```
   > # 예시
   > getUser(id: number): User
   >
   > 인터페이스는 단순히 number를 받고, User를 반환한다는 정보만 전달하지만 협약은 조건을 추가함
   >
   > ✔ 선행조건 예시
   > - id는 1 이상이어야 한다
   > - 호출자는 null을 전달해서는 안된다
   > - 데이터베이스 연결은 유효해야 한다
   >
   > ✔ 결과조건 예시
   > - 반환되는 User.id는 전달된 id와 같아야 한다
   > - 존재하지 않는 id일 경우 예외를 던진다
   > - User 객체는 null이 될 수 없다
   >
   > ✔ 불변조건 예시
   > - User 객체는 id가 변경되지 않아야 한다
   > - 내부 캐시는 항상 정렬된 상태를 유지해야 한다
   > ```
   > {: .prompt-info }

7. 재사용 가능한 소프트웨어 요소

   - 소프트웨어 부분, 전체 영역 모두 재사용 요소
   - 코드뿐 아니라 응용된 지식, 데이터 구조 등도 재사용 요소 (개발 이후 테스트 계획, 문서화 방법 및 절차 등)

8. 소프트웨어 재사용 방법

   - 합성 중심: 모듈(블록)을 조립하는 블록 구성 방식
   - 생성 중심: 추상적 명세(기본적인 틀)를 구체화하는 패턴 구성 방식 

<br>

## 2. 소프트웨어 개발 수명 주기 

### 1. 소프트웨어 개발 수명 주기

1. 소프트웨어 개발 수명 주기 정의

   - 소프트웨어 개발 방법론의 바탕
   - 개발 과정을 단계별로 정의 (단계별 산출물 존재)

2. 폭포수 모델

   - 과거에 폭넓게 사용되던 방식
   - 정해진 단계를 한 번씩만 진행하며 **이전 단계로 돌아갈 수 없음**
   - 단계별로 결과물이 명확해야 다음 단계로 넘어감
   - 제품의 기능 보완이 불가능하므로 **메뉴얼 작성이 필수**

   ![waterfall.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/waterfall.jpg)
   _폭포수 모델_

3. 프로토타입 모델

   - 폭포수 모델의 단점을 보완한 모델
   - **시제품**을 통해 최종 결과물을 예측
   - 시제품은 사용자와 시스템 사이의 **인터페이스에 중점**을 두어 개발
   - 시제품은 추후 최종 구현 단계에서 **골격**으로 사용

   ![prototype.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/prototype.jpg)
   _프로토타입 모델_

4. 나선형 모델

   - (폭포수 모델, 프로토타입) 모델에 **위험 분석**을 더한 기능
   - 여러 번의 지속적인 개발 과정을 통해 점진적으로 개발 (사이클을 통해 완성도가 올라감)
   - 개발 중 발생할 수 있는 **위험을 최소화하는 것**이 목적이며 **유지보수가 필요없음**
   - 누락 및 추가된 요구사항 반영이 가능

   ![spiral.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/spiral.jpg)
   _나선형 모델_

5. 애자일 모델

   - 고객과 **소통에 중심**을 둔 방법론의 통칭
   - 짧은 개발 주기를 반복하며 고객의 피드백 반영
   - 고객과의 소통을 통해 작업의 우선순위를 지정
   - **소통, 협업, 변화 대응**에 가치를 둠 (절차, 문서, 계획 X)
   - 애자일 기반 모델: **Scrum, XP**, Kanban, crystal, FDD(기능 주도 개발), ASD(적응형 소프트웨어 개발), DSDM(동적 시스템 개발)

<br>

### 2. 스크럼 모델

1. 스크럼 모델 특징

   - 스크럼 팀을 구성, **팀을 중심**으로 개발의 효율성을 높임
   - 제품 책임자, 스크럼 마스터, 개발팀으로 구성
   - **확약, 전념, 정직, 존중, 용기**에 가치를 둠
   - 반복적인 스프린트를 통해 제품을 완성

   ```
   - 스프린트: 2~4주 정도의 기간 내에서 하나의 task를 개발하는 과정
   - 태스크(task): 개발 요구사항(사용자 스토리)을 개발자(팀)별로 나눈 것
   ```

2. 제품 책임자

   - 목표 제품에 대한 책임을 지고 의사를 결정하는 역할 담당
   - 이해관계자(stakeholder)들의 의견을 종합하여 요구사항을 백로그에 작성하고 우선순위를 조정
   - 팀원들은 백로그에 스토리 추가만 가능하고 우선순위 조정은 불가능

   ```
   - 제품 백로그: 우선순위에 따라 사용자 스토리를 나열한 목록
   - 스프린트 백로그: 해당 스프린트에서 개발해야 할 태스크를 나열한 목록
   - 사용자 스토리: 사용자 요구사항을 이야기(시나리오)의 형태로 표현한 것
   - 릴리즈 계획: 제품 백로그에 작성된 사용자 스토리를 기반으로 전체 개발 일정을 수립 
   ```

3. 스크럼 마스터

   - 개발팀들의 업무를 위한 가이드 역할 담당
   - 일일 스크럼 회의를 주관하며, 개발 과정에서 발생한 장애 요소를 공론화하여 해결할 수 있도록 처리
   - 팀원들이 상황에 유연하게 대응할 수 있도록 조력하는 역할 (통제 X)

4. 개발팀

   - 제품 책임자, 스크럼 마스터를 제외한 개발에 참여하는 모든 인원 (개발자, 디자이너, 테스터 등)
   - 능동적으로 팀을 구성하고 문제를 해결

5. 스크럼 모델 개발 프로세스

   - 스프린트 계획 회의: 스프린트 백로그 작성, 개발 일정 수립
   - 스프린트 진행: 개발자에게 태스크 할당
   - 일일 스크럼 회의: 소멸차트를 통해 진행 상황 점검, 태스크 진행 상황에 따라 (할 일, 진행 중, 완료)의 상태로 변경
   - 스프린트 검토 회의: 개발이 완료된 부분 또는 전체 제품을 사용자와 함께 테스트, 사용자 피드백을 제품 백로그에 반영
   - 스프린트 회고: 스프린트 진행 자체의 문제점, 개선점 도출

   ![scrumProcess.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/scrumProcess.jpg)
   _스크럼 모델_

<br>

### 3. XP(eXtreme Programming) 모델

1. XP 모델 특징

   - **고객의 참여, 짧은 개발 과정의 반복**을 극대화하여 개발 생산성 향상
   - 단계별 단순한 설계를 통해 개발 속도 향상
   - **소규모 인원**의 프로젝트에 효과적
   - **의사소통, 단순성, 용기, 존중, 피드백**에 가치를 둠

2. XP 모델 개발 프로세스

   - 사용자 스토리에 기록용 내용을 바탕으로 릴리즈 계획 수립
   - 분석된 스토리에 따라 스파이크 또는 이터레이션 진행
   - 스파이크를 통해 기술이 검증되면 해당 부분을 이터레이션으로 전달
   - 이터레이션 진행 도중 새로운 스토리가 작성될 수 있음
   - 이터레이션을 통해 부분 완료된 제품을 고객이 직접 사용자 스토리에 포함된 테스트 사항을 통해 승인 검사 수행
   - 테스트 과정에서 새로운 요구사항, 오류 등이 발견되면 다음 이터레이션에 반영

   ```
   - 소규모 릴리즈: 기능별 사용자 피드백을 받을 수 있도록 릴리즈 규모를 작게 분할
   - 스파이크: 특정 기술에 대한 검증만을 위해 개발하는 프로그램 (다른 요소는 모두 배제)
   - 이터레이션: 하나의 릴리즈를 1~3주의 개발 기간으로 세분화한 단위
   ```

   ![xpProcess.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/xpProcess.jpg)
   _XP 모델_

3. XP 기본 원리

   ```

   - Continuous Intergration

   - Design Improvement

   - Pair Programming
   - Planning Game

   - Simple Design
   - Small Releases
   - Sustainable Pace
   - System Metaphor

## 3. 소프트웨어 개발 방법론

### 1. 소프트웨어 개발 방법론

1. 소프트웨어 개발 방법론 정의

   - 소프트웨어 개발 전 과정에 지속적으로 적용 가능한 방법, 절차, 기법
   - 소프트웨어 품질과 생산성 향상 (= 효율적으로 개발하는 법)
   - 소프트웨어 공학에서 가장 많이 활용하는 방법론

2. 소프트웨어 개발 방법론의 절차

   - 분석: 개발 준비, (시스템 및 소프트웨어 요구사항) 분석
   - 설계: (시스템, 소프트웨어 구조 및 상세) 설계
   - 구현: (소프트웨어 코딩 및 단위) 시험
   - 시험: 시스템과 소프트웨어 통합 및 테스트, 소프트웨어 설치 및 인수 지원 

3. 소프트웨어 개발 방법론 선정

   - 개발 방법론의 특징을 고려하여 타당성과 적절성 설정
   - 개발 방법론 선정을 위한 계획서 작성
   - 계획서를 바탕으로 정성, 정량평가를 진행하여 개발 방법론 선정
   ```
   - 타당성: 개발 절차에 따라 설정 
           (요구사항 분석 절차가 목표 시스템을 개발하기 적절한가)

   - 적정성: 개발 단계별 산출물에 따라 설정
           (요구사당 분석에 의한 산출물이 목표 시스템에 필요한 실제 산출물과 일치하는가)
   ```

<br>

### 2. 소프트웨어 개발 방법론 종류

1. 구조적 방법론

   - **Yourdon**이 개발, 1970년대까지 가장 많이 적용
   - 구조적 분석을 통해 고객의 요구사항을 **자료 흐름도**로 표현
   - **모듈 중심의 설계**를 통해 모듈 간 결합도를 낮춰 독립성를 높임
   - **순차, 선택, 반복**의 논리 구조 구성으로 프로그램 복잡성을 최소화

   ```
   - 자료 흐름도(DFD): 프로그램을 기능 단위별 데이터의 흐름으로 표현한 구조도
   - 자료 사전(DD): 자료 흐름도(DFD)에 표현된 자료 저장소를 구체화
   ```

   ![structured.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/structured.jpg)
   _구조적 방법론_

2. 정보공학 방법론

   - 개발 단계별 정형화된 기법들을 통합 적용한 데이터 중심 방법론, 1980년대 등장
   - 현행 업무 프로세스 및 시스템을 분석하여 정보 전략 계획 수립
   - 업무 영역 분석을 통해 개념적인 수준의 데이터와 프로세스를 설계
   - ERD를 기반으로 분할, 액션, 의존 다이어그램 등을 활용해 실질적인 시스템 설계

   ```
   - 데이터 모델링 도구: 개체-관계 다이어그램(ERD)
   - 프로세스 모델링 도구: 자료 흐름도(DFD), 프로세스 의존도(PDD), 프로세스 계층도(PHD)
   ```

   ![informationEngineer.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/informationEngineer.jpg)
   _정보공학 방법론_

3. 객체지향 방법론

   - **실체**(Entity)를 독립된 형태의 **객체**(Object)로 표현
   - 객체들 간 메세지 교환을 통해 상호작용하도록 개발

   ```
   - 속성: 객체를 나타내는 성질, 값, 데이터
   - 메서드: 객체의 속성을 이용한 일련의 동작들
   ```

   - 데이터 객체를 저장하기 위해서 **관계형 테이블**로 변환하는 과정(Object-Relation Mapping)이 필요

   ```
   객체지향 방법론의 기본 원칙

   - 캡슐화: 데이터와 해당 데이터의 처리 기능을 하나로 묶어냄
   - 정보은닉: 다른 객체에게 자신의 정보를 숨김
   - 추상화: 객체의 공통적인 속성을 상위 객체로 도출
   - 상속성: 상위 객체의 속성을 하위 객체가 물려받아 사용
   - 다형성: 하나의 수행 방법으로 여러 형태의 기능 수행
   ```

   (_객체지향 방법론_) <br>
   요구분석 - 객체지향 분석 - 객체지향 설계 - 객체지향 구현 - 데이터 객체 / 기능 객체
      

4. 컴포넌트 기반 방법론 (CBD: Component Based Development)

   - **컴포넌트를 조립**해서 개발하는 방법론
   - 개발 생산성, 이식 및 호환성, 신속성, 유연성, 표준화 등의 장점
   - **선행 투자 비용이 높고**, 조립식 시스템에 따르는 **책임 및 지적 재산권을 고려**해야함
   - 일반적으로 프로세스 설계 과정에선 객체지향 방법론, 데이터 설계 과정에선 정보공학 방법론 사용

   ```
   컴포넌트 기반의 단계별 산출물

   - 분석: 사용자 요구사항 정의서, 유스케이스 명세서, 요구사항 추적표
   - 설계: ~설계서, ~계획서, ~기술서, ~케이스, 시스템/통합시험 시나리오
   - 구현: 프로그램 코드, 단위 시험 결과서, 데이터베이스 테이블 
   - 시험: ~결과서, ~지침서, 인수시험 시나리오
   ```

   ![componentBased.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/componentBased.jpg)
   _컴포넌트 기반 방법론_

5. 애자일 방법론

   - 수시로 변하는 상황, 고객의 요구사항을 즉시 반영하여 개발

6. 제품 계열 방법론

   - 특정 제품에 적용하고 싶은 공통된 기능을 개발
   - **임베디드 소프트웨어**를 작성하는데 유용

   ```
   - 영역공학: 영역 분석, 영역 설계, 핵심 자산을 구현하는 영역
   - 응용공학: 제품 요구 분석, 제품 설계, 제품을 구현하는 영역

   (두 공학을 연계하기 위한 제품 요구사항, 제품 아키텍처, 제품 조립 생산이 필요)
   ```

   ![productLine.jpg](/assets/img/posts/Certifications/Engineer-Information-Processing/2025-12-08-SoftwareDevelopmentMethodology/productLine.jpg)
   _제품 계열 방법론_

<br>

### 3. 소프트웨어 보안 개발 방법론

1. 보안 개발 방법론 정의

   - 보안 취약점을 최소화하기 위한 지침 및 사례를 기반으로 개발
   - 시스템 환경에 따라 취약점이 달라 다양한 보안 방법론이 존재

2. MS-SDL

   - 마이크로소프트사라 자체적으로 수립한 모델

3. Seven Touchpoints

   - 실무적으로 검증된 보안의 모범 사례 7가지를 개발 모델에 통합

   ```
   - 코드 검토
   - 아키텍처 위험 분석
   - 침투 테스트
   - 위험 기반 보안 테스트
   - 악용 사례
   - 보안 요구
   - 보안 운영
   ```

4. CLAPS (Comprehensive, Lightweight Application Security Process)

   - 소프트웨어 개발 생명 주기(SDLC)에 걸친 포괄적으로 효율적인 보안 모델
   - 활동 중심, 역할 기반의 프로세스로 구성
   - **이미 운영 중인 시스템에 적용하기 좋음**
   - **개념, 활동 평가, 활동 구현, 역할 기반, 취약성**의 5가지 관점에 따라 보안 절차 진행

5. CWE (Common Weakness Enumeration)

   - 보안 취약점을 체계적으로 분류한 데이터베이스 <br>
   (취약점 원인을 7가지로 정리, Seven touchpoints와 다름)
